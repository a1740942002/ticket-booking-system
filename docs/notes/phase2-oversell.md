# Phase 2 學習筆記 — 見證超賣（Race Condition / Lost Update）

日期: 2026-05-30
相關測試: `apps/api/test/concurrency.test.ts`
相關程式: `apps/api/src/routes/orders.ts`（天真扣庫存,第 15–23 行）

## 實驗

100 個人同時搶一個只有 **10 張票** 的票區（`Promise.all` 同時發出 100 個 `POST /orders`）。
正確的系統最多應該只賣出 10 張。

## 結果

```
下單成功 (200):       100      ← 100 人全部搶到
被擋下 (409 缺貨):    0
實際建立的訂單數:     100      ← 只有 10 張票卻建了 100 筆訂單
剩餘庫存 availableQty: 9       ← 只少了 1！
總票數 totalQuantity:  10
超賣張數:             90
```

測試斷言「售出 ≤ 庫存」失敗（Expected `<= 10`, Received `100`）。

## 為什麼剩餘庫存是 9，而不是 0 或 -90？

關鍵在 `orders.ts` 那段「讀 → 改 → 寫」在 100 個請求間**交錯執行**:

```
請求#1:  讀到 available=10  →  檢查 10>=1 ✓  →  算出 9  →  寫回 9
請求#2:  讀到 available=10  →  檢查 10>=1 ✓  →  算出 9  →  寫回 9
   ...   (100 個請求幾乎同時讀到「還有 10」)
請求#100: 讀到 available=10  →  檢查 10>=1 ✓  →  算出 9  →  寫回 9
```

因為**所有人都在任何一次寫入「生效」之前就先讀到了 `10`**:

1. **檢查全部通過** — 每個請求都覺得「10 ≥ 1，庫存夠」，所以沒人被 409 擋下 → 100 筆訂單全建立。
2. **寫回互相覆蓋** — 每個請求都把 `10 - 1 = 9` 寫回去，後寫的蓋掉先寫的。最後庫存停在 `9`，只「少了 1 張」。

## 兩個經典併發 bug

- **Race condition（競態）**:最終結果取決於讀寫交錯的時序，不是程式邏輯本身。
- **Lost update（更新遺失）**:99 次扣減被覆蓋掉，等於憑空消失。

> ⚠️ 重點:`availableQuantity` 甚至沒變負數，反而更危險 —— 光看「剩 9」完全看不出已經超賣 90 張。
> 教訓:**「讀出來的值，在你寫回去的瞬間可能早就過期了。」** 任何「先讀、在記憶體裡判斷、再寫回」的庫存扣減，在併發下都不可靠。

## Phase 3 的解法方向

核心思路:把「檢查 + 扣減」變成**單一原子操作**，別在應用程式記憶體裡做決定。
依序示範三招，每招都重跑此測試看它轉綠並比較優劣:

1. **原子條件更新** — `UPDATE ... SET available = available - 1 WHERE available >= 1`，由資料庫保證原子性（最簡單有效）。
2. **悲觀鎖** — `SELECT ... FOR UPDATE` 鎖住該列，序列化存取。
3. **樂觀鎖** — `version` 欄位 + 失敗重試（compare-and-swap）。
